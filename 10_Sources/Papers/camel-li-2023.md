# CAMEL: Communicative Agents for "Mind" Exploration of Large Language Model Society

> Full citation: Guohao Li, Hasan Abed Al Kader Hammoud, Hani Itani, Dmitrii Khizbullin, and Bernard Ghanem. "CAMEL: Communicative Agents for 'Mind' Exploration of Large Language Model Society." arXiv, 2023. https://arxiv.org/abs/2303.17760

## Summary

The paper addresses a gap in chat-based large language models: task completion still depends on a human supplying a stream of precise prompts. The authors propose role-playing, a framework in which two instances of a language model, an AI assistant and an AI user, cooperate on a task from a single human-supplied idea, with no further human steering. A task-specifier agent turns the idea into a concrete task, then the assistant and user exchange instructions and solutions in a loop until the user emits an end-of-task token. The authors call the system prompts that set this up inception prompting, since the agents prompt each other automatically once the conversation starts. They use the framework to generate four large conversational datasets (AI Society, Code, Math, Science) at scale, and show that solutions produced by CAMEL role-playing beat single-shot gpt-3.5-turbo answers in both human and GPT-4 evaluation, and that fine-tuning LLaMA-7B on the generated data yields measurable gains on HumanEval and HumanEval+.

## Key Concepts

- Role-playing: a task-oriented cooperative framework with exactly two agents, an AI assistant and an AI user, each bound to a fixed role for the session.
- Inception prompting: the one-time prompt engineering done before the conversation starts, comprising a task-specifier prompt, an assistant system prompt, and a user system prompt, after which the agents drive each other without further human input.
- Task specification: an automatic step in which an LLM agent expands a vague human idea (for example, "develop a trading bot for the stock market") into a specific, actionable task.
- Critic-in-the-loop: an optional agent, human or AI, that selects among or gives feedback on proposals, enabling tree-search-like decision making during task-solving.
- CAMEL_TASK_DONE: an end-of-task token the user agent emits to terminate the conversation once it judges the task complete.

## Terminology

- AI assistant agent: the role that executes instructions and returns a "Solution:" formatted response, ending each turn with "Next request."
- AI user agent: the role that issues instructions (with or without an input field) and drives the conversation toward completion.
- Role flipping: a failure mode where the assistant starts issuing instructions and the user starts following them, reversing the intended roles.
- Flake reply: a non-committal assistant response such as "I will..." that fails to deliver an actual solution.
- Message-passing graph: the paper's generalisation of the two-agent formulation to model communication among more than two agents, or between humans and AI.

## Architecture and Implementation

A role-playing session starts from a human-supplied idea and a pair of role labels (for example, Python Programmer and Stock Trader). A task-specifier agent turns the idea into a specific task description. Two system prompts, one for the assistant and one for the user, are then issued once, each declaring the agent's role, the shared task, communication rules ("never flip roles," "never instruct me"), a response format, and termination behaviour. From that point, the user agent proposes an instruction (with an optional input field) and the assistant agent responds with a "Solution:" turn, and the pair repeats this loop, with the conversation history passed back into each turn, until the user signals completion or a termination condition fires. Termination conditions include no instruction from the user for three rounds, an apparent role reversal, the end-of-task token, hitting the model token limit, or a cap of forty messages, imposed because conversation cost grows quadratically with length. For the AI Society and Code datasets, roles and tasks are themselves generated by prompting an LLM, rather than supplied by a human, which is what makes the pipeline scale to 25,000 conversations for AI Society alone (50 assistant roles by 50 user roles by 10 tasks each).

## Best Practices

- State the shared task and both role names in the system prompt for each agent, and repeat "never forget" framing to keep the model anchored to its role across a long conversation.
- Forbid the assistant from asking questions, since the authors found this a direct trigger for role flipping.
- Require a fixed response format ("Solution: ... Next request.") to stop flake replies and keep instruction-solution pairs usable as fine-tuning data.
- Give the user agent an explicit, single end-of-task token, since without one the paper observed agents falling into indefinite loops of pleasantries.
- Set hard termination conditions (message cap, token limit, no-instruction rounds) because unattended agent conversations do not reliably self-terminate.

## Warnings and Anti-Patterns

- Role flipping: the assistant can drift into issuing instructions, taking over the user's role, especially when it is allowed to ask questions.
- Assistant repeating instructions: the assistant sometimes echoes the user's instruction back rather than solving it.
- Infinite loops: agents can become stuck exchanging thanks or farewells, occasionally while apparently aware they are stuck, without breaking out unaided.
- Misalignment risk: the authors generated a dedicated Misalignment dataset to illustrate how an unaligned instance of the same framework could simulate malicious cooperative tasks, flagging this as a real risk of autonomous multi-agent systems rather than a hypothetical one.

## Future Work

The authors position CAMEL as a foundation for future work in multi-agent systems, cooperative AI, game-theoretic simulation, social analysis, AI ethics, and AI alignment. They flag the challenges they catalogue, role flipping, flake replies, infinite loops, and conversation deviation, as open problems for achieving reliable autonomous cooperation, and note their critic-in-the-loop mechanism as an initial, not fully explored, direction for improving controllability.
